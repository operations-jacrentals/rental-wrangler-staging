Compose & comms rail · spec §7.3 · matte, no glow · first draft for reaction

Compose docks — it never teleports you

A comms action fired from a record — right-click, +Team, +Gmail, Text…, Email… — opens the new message where you are: a minimizable footer tab on desktop (Gmail-desktop's docked compose, several at once), a full-screen sheet on mobile. It never navigates you to the Inbox card. Three compose types duplicate three best-in-class feels in RW's own skin: Gmail (email), Apple Messages (text), Facebook Messenger (team) — not the incumbent comms rail.

Same action, coarse-pointer target: the composer takes the whole screen instead of a footer sliver (a 350px docked panel doesn't fit a phone, and stacking tabs would fight the keyboard). One ✕ takes you back to exactly the card + row you were on — never the Inbox.

Mobile ruleThe composer reuses the comms full-screen gesture — the same surface the app already uses elsewhere for a focused mobile mode, not a bespoke overlay. Closing (✕) always returns to the record you were on, mid-scroll-position intact.
No stacking on phoneOnly one compose lives at a time on mobile — there's no room for docked tabs, so starting a second message replaces the first (draft kept, resumable from its record or the Inbox).
Judgment calls in this draft: the R20 menu keeps the generic Copy/Global-Search items above a divider so this reads as the real menu, not a stripped demo, with the four comms actions grouped under a "re: Gator / RC-4700" heading that names what's riding along. "+Gmail" is wired to a blank To/Subject (the generic compose) while "Text/Email Denny…" pre-fill — showing both compose starts side by side. Only one dock item is ever expanded at once (Gmail's multi-window compose was intentionally simplified) — a minimized tab's unread dot borrows Signal's fill=today colour logic. The pop-out icon in each dock header is inert here (title-only) — it's the pointer to "open in Inbox," not built out since the Inbox card is a separate mock.
