Judgment calls

  • Real stages, sourced verbatim from config.js FUNNELS: rental ['Lead','Reserved','Rented'], member ['Lead','Contacted','Not A No!','Payment Discussed','Signed'] (auto: Signed), equipment ['Lead','Contacted','Not A No!','Payment Discussed','Paid'] (no auto yet — flagged in the code itself for a future auto-trigger). The Rental tab renders Rental's own 3 stages plus Member's 5 as a second stacked ladder when Member Lead is ticked — matching funnelSectionHtml's rentalBody() exactly (two independent Lead entries, not one shared one — the earlier 2026-07-17 design doc proposed a single shared Lead rung, but the shipped code keeps them separate, so I followed the code).
  • Icons are reused verbatim from the app's own GATE_ICON (app.js ~16845) — Reserved (calendar), Contacted (phone), Not A No! (thumbs-up), Payment Discussed (card), Paid (wallet) map 1:1 by stage name. Three stages have no exact GATE_ICON key today, so I reused the closest real semantic match rather than inventing one: Lead → the 'Inbound Lead' glyph, Rented → the 'On Rent' glyph (same real-world state, different label), Signed → the 'Complete' glyph (agreement completed). Flagging these three for Jac to confirm or swap.
  • Done/current/upcoming no longer differ by icon (the icon is now the stage's fixed identity) — the state reads instead through fill vs. outline (a shape cue, not just colour), plus a small corner badge: a circle ✓ for done, a diamond (◆, pure geometry, no colour dependency) for current, no badge for upcoming — on top of the ring and "· now" text already carrying "current." Four independent non-colour cues on the one state that matters most.
  • Per-checkpoint actions model the app's real 3-state layer logic (funnelLayerHtml): a stage with an open, dated action shows that action's text + urgency (a filled-yellow Signal only when due today, else a quiet Stamp) + a commit Door; an unarmed stage (done or upcoming) shows a ghost + Action. The one auto-locked next stage in this sample (Rental's Payment Discussed → Signed) renders its Gate popover as an informational note instead of a clickable row, since reachFunnelStage genuinely refuses to manually set an auto stage — the mockup doesn't offer a control the real app would reject.
  • "More Actions" is the overflow — account-level items with no fkey/stage tag, distinct from the per-checkpoint actions now inline on the stepper (the real nextActionsHtml currently lists both redundantly; this mockup splits them for clarity).
  • Inline add, no modal: "+ Action" expands a row in place (text field + Save/Cancel) rather than opening a window. The one tolerable popup is the small calendar + time picker, which opens anchored above the clicked trigger (falls back to below only if there's truly no room above — an edge-case safeguard, not the default). Saving swaps the ghost button for the armed display live, so the whole loop is demonstrable without a page reload.
  • Toggle rule applied: the active tab is a filled Signal of that funnel's own state — yellow "· today" when a today-action exists anywhere in it, else the standard orange (no real status = neutral). Panel A's Rental tab demonstrates the yellow branch; Panel B's Equipment tab (same customer, same data) demonstrates the orange fallback, since Equipment carries no due-today item. Inactive tabs keep a small dot instead of the full fill.
  • Yellow dimmed to #eed44b in BOTH themes to match the locked wrangler-style token — light theme no longer keeps the brighter #ffe14d. The dark-palette deuter/protan separation (≥90 vs orange & green) was verified when the hex was locked; light-theme separation holds by lightness (the yellow is far lighter than light-orange #cf6000 / green #0c8f5f).
  • Control height moved 27px → 26px to match the app's own real precedent in this exact section (h:26 on addBtn/actionPill calls in funnelSectionHtml's neighborhood); chip radius stays unified at 7px (style's two-radii rule still overrides wrangler-style's separate 6/8 listing).
