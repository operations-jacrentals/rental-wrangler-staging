Inbox v3 — channels up top, talk native

Two structural changes from v1. One — channel moves out of the buried left-rail "Streams" list and up to a top toggle: ALL · TEAM · TEXTS · EMAIL · WRANGLER · CALLS, each carrying its own unread count, CALLS dimmed as a "soon" placeholder. The left rail goes back to being just Gmail's folders. Two — a thread now renders in its medium's own native form instead of Gmail's reading pane wearing every hat: email stays a Gmail thread, a text becomes Apple-Messages bubbles, a Team post becomes a Messenger channel with presence + reactions, and Mr. Wrangler gets his own bot-thread with a loud, distinct bell when he actually fixes something. The chrome around all of it — search, filters, sort — is ours, not Gmail's: the mini-search bar with pills live in it, the globe scope toggle, and a Views & sort menu. Same cross-record continuity as v1 — every thread still knows which customer/rental/unit it's about.

v3 adds four things from spec §7.4–§7.5, all new sections below (7–10): the card's own responsive space model — three width stages plus the left-menu hover/click choreography that swaps the hamburger menu for the thread list without ever moving it; an Inbox-only Focus toggle (⅓ → ½ → full) that shrinks its two siblings to slim clickable spines and rebalances the instant you touch one; a shared quick-action set — Hide · Mark unread · Star · Snooze · Archive — live as row actions and a right-click menu on the same five; and the inbound-routing rule for where a new text or email actually lands. Interactive throughout — hover, click, and right-click the demos, don't just read them.

8 Inbox expand — Focus toggle + sibling spines (spec §7, inbox-only)

The Inbox is the only card that can grow at its siblings' expense. A stepped Focus toggle (⅓ → ½ → full) lives on the Inbox card only; as it grows, the other two cards collapse to slim clickable spines — title + an unread/alert dot, nothing else. Hover a spine for a transient peek (nothing commits); click it and the state machine is exactly {balanced} ↔ {inbox-focused} — the Inbox drops back to ⅓ and that card comes forward, ready to work. There is never a "both expanded" state.

Judgment calls made for this v2 pass, flagged so Jac can push back:
  • Boudreaux Marine's thread splits into two native, channel-tagged threads (email + texts) instead of staying one interleaved Gmail thread — the direct consequence of "channel is a top-level toggle, native per medium." Both keep the same Boudreaux Marine LLC / Gator RC-4700 Ref chips, and the email side now explicitly nods to the parallel text thread, so the "one relationship" story still reads even though it's told across two rows.
  • Fixed a pre-existing v1 contrast bug while cleaning up pure #fff: light theme's --onAcc was pure white on the mid-orange accent (3.88:1 — fails AA), which also contradicted wrangler-style's own "orange fill = dark ink ALWAYS" rule. Now dark ink in both themes (4.70:1). Light --panel/--card moved off pure white to #fbfbfa; the four remaining hardcoded #fff inks (red-fill/commit text, badge, FAB) now route through a new --onFill near-white token instead.
  • Filter pills inside the mini-search get pill radius (999), not the row's chip radius (7) — reads as "removable action," kept visually distinct from the Signal/Ref/Stamp chips sitting in the same view.
  • CALLS is genuinely inert (pointer-events:none, dimmed, a "Soon" marker) rather than just styled to look disabled — WCAG's non-text-contrast floor explicitly exempts inactive UI components, so it's dimmed harder than any live control would be.
  • Reactions render as Lucide heart/laugh icons, not emoji — keeps the "inline SVG icons only" rule intact; a real build might prefer native emoji for Messenger-authenticity, worth Jac's call.
  • Views & sort is visual-only this pass (sort options + two saved Views, no live query behind them) — live sort/filter logic is the parked "Sort redesign" from the spec's Open Problems, not this slice.
  • Recent-searches-above is demonstrated via DOM order (a column-reverse flex wrapper), not a true floating overlay — a real build would position:absolute it so it doesn't push layout; the visual direction ("opens above, never below") is faithful either way.
  • Mobile channel swipe is a real gesture (touchstart/touchend, ~40px threshold, plus tappable chevrons), wraps at both ends, and re-filters the same thread rows live; landing on Calls shows a "coming soon" placeholder instead of an empty or broken list.

v3 additions — judgment calls:

  • Fixed the last 2 pure-black instances — --shadowPop's dark-theme value was literally rgba(0,0,0,.45) (twice: the :root default and the explicit data-theme="dark" block). Both now use rgba(5,7,10,.45) — the same near-black already established by this file's own .scrim (rgba(5,7,10,.55)), so it's not a new tone, just consistent with one already in use.
  • One-card choreography is built as two fixed slots whose CONTENT swaps (slot 1: list ↔ pinned menu; slot 2: email ↔ wide list), not as DOM nodes physically moved between panes — same visual result ("list slides into the email's place, menu pins left") without a manual FLIP animation. The hover-peek overlay is pure CSS (a sibling-combinator hover rule), no JS needed for that half of the choreography.
  • Inbox-expand's spines use writing-mode:vertical-rl for the collapsed title — reads as a genuine slim, clickable spine (like a collapsed IDE panel) rather than just a narrow box with truncated text. Clicking a spine rebalances via the exact {balanced}↔{inbox-focused} toggle the spec calls for; the "sibling comes forward" cue is a brief accent-border flash (reusing the existing .flash pattern from §1's panel), not a new component.
  • Two new Lucide glyphs added to the sprite — eye-off (Hide) and grip-vertical (drag handle) — both standard Lucide icons, traced faithfully, not hand-rolled. Snooze reuses the existing clock glyph (already doing double duty as "recent" elsewhere in this file) rather than adding a third clock-ish icon.
  • Quick actions are wired generically onto the real §3 workspace list via a small JS injector (wireRowActions) rather than hand-duplicating the hover/right-click markup onto all nine existing rows — same five actions, same icons, same order, everywhere; §9 also carries a static legend + a standalone swipe-reveal demo for the mobile case, and Star isn't hover-duplicated since every row already has its own always-visible star control (the right-click menu still lists it, for parity).
  • Drag-to-resort is a hinted affordance only (a grip icon that fades in on hover), per the brief — no functional drag-and-drop reordering in this mock.
  • Inbound routing gets its own compact dock (§10) rather than reusing the §6 dock demo's live DOM nodes — keeps the simulator self-contained so Jac doesn't have to scroll back up mid-demo; same .dockchip/.composedock vocabulary throughout.
  • All new v3 elements reuse existing tokens only — no new hex values were introduced, so every new text/fill pairing (spine titles, row-action icons, context menu, dock preview, swipe-action fills) is a already-checked combination borrowed verbatim from elsewhere in this file (e.g. the swipe-action ink colours are the same #211a05/#04140d already used on filled yellow/green Signals).
