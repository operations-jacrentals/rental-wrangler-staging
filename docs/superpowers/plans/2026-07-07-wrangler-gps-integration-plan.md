## Two tracks, run in parallel

Phase 1 splits into an **infra/ops track** (not code in this repo) and a **frontend track** (`app.js`/`config.js`/`style.css`). The frontend can be built and gate-tested against a **mock backend** before the real service is live, so the two tracks don't block each other until the Staging E2E.

### Track A — Infra/ops (outside this repo; Jac + a Haiku/plumbing pass where scriptable)

A1. **Stand up our own backend.** New Railway project under JacRentals' account; deploy the forked `operations-jacrentals/wranglergps` `backend/` unchanged; provision Railway Postgres (`DATABASE_URL`); set fresh `SESSION_SECRET` + `DASHBOARD_PASSWORD`; set the four providers' credentials (already JacRentals-owned) and the Deere/Bouncie `*_REDIRECT_URI` to the new Railway URL. Reconnect Deere + Bouncie OAuth once via Settings to seed tokens (README runbook).
A2. **CORS/allowed origin.** Set `FRONTEND_URL` to `https://app.jacrentals.com`; also allow the staging mirror origin for the E2E step.
A3. **~~New backend endpoint `GET /api/fleet/status`~~ — DROPPED (2026-07-07).** Reading `frontend/src/hooks/useFleet.js` showed the four-provider merge is already done **client-side** by calling the existing per-provider endpoints (`/api/hapn/devices` + `/fleet-status`, `/api/deere/status`+`/machines`, `/api/yanmar/status`+`/machines`, `/api/bouncie/status`+`/vehicles`) with `Promise.allSettled` (partial-failure tolerant). Rather than add a new endpoint to the friend's already-debugged backend (which the README warns against touching), **Track B mirrors this merge in our own `app.js`** — we control it, and the vendored backend stays untouched. Net: **zero new backend routes.** The only fork change is A3b.
A3b. **CORS — allow our origins.** DONE — fork PR operations-jacrentals/wranglergps#1 (draft): adds `.github.io` (staging mirror) + `localhost:9147` (local dev) and makes `FRONTEND_URL` a comma-list. Merges when Railway is stood up.
A3c. **NEW (2026-07-07) — `device_events` table + two routes, on the fork.** The connect wizard and the history feed (spec §5a/§6a) need durable, server-side event storage that the browser cannot provide. Additive to the forked backend, same pattern as A3b (a small fork PR, deploys on next Railway push, no existing route touched):
   - `backend/db.js`: new table `device_events (id, source, device_key, type, detail JSONB, by, at)` — `type` ∈ `status_change` | `alert` | `shutdown_command`.
   - `backend/routes/hapn.js` (or a new small `routes/events.js`): the existing starter-interrupt toggle route also writes a `shutdown_command` row (outcome + who) on every call — piggyback, don't duplicate the toggle logic.
   - New route `GET /api/device/:source/:key/events` — reads `device_events` for one device, used by the history feed (§6a) on GPS-section open.
   - Status-transition detection piggybacks the backend's *existing* crons (hourly Hapn backfill, Bouncie/Deere keep-alives) rather than adding new polling — compare last-known status to current on each cycle, write a row only on change.
A4. **Unit↔tracker backfill (ops task).** Walk the fleet, running the connect wizard (§5a) per unit to record `gpsProvider` + `gpsDeviceId` with confirmed first contact. Blocks *meaningful* data per unit but not the frontend build.

### Track B — Frontend (this repo, `area/wrangler-gps`)

Build order — each step ends green before the next.

1. **Config + client module.** Add `GPS_BACKEND_URL` to `config.js` (alongside `BACKEND_URL`; a referrer-safe public URL). Add a small `gps` client in `app.js` (new APP chapter banner — update the code-map): `gpsLogin()` (calls `/auth/login` with the team password on Rental Wrangler login, caches token in memory), `gpsFleetStatus()` (client-side merge across the four providers' existing endpoints, per A3's dropped-endpoint decision), `gpsUnitStatus(provider, deviceId)` (live detail), `gpsShutdown(deviceId)` (Hapn starter-interrupt), `gpsDeviceEvents(source, key)` (reads the new A3c history endpoint), `gpsProviderDevices(provider)` (Deere/Yanmar/Bouncie machine-list lookups, for the wizard's picker step). All wrapped in the existing timeout helper (app.js:15923) and a try/catch that degrades gracefully. **No proxy through Apps Script** — direct fetch, per spec §4.
2. **Connect-a-device wizard (spec §5a) — replaces the old plain-text mapping fields.** A small popup launched from the GPS section's "No GPS" badge/edit affordance: (a) provider segmented control, (b) Hapn → IMEI text field, Deere/Yanmar/Bouncie → searchable list from `gpsProviderDevices()`, (c) live poll loop (`waiting for signal…` → `device seen…` → `✓ Reporting`, capped attempts, explicit "not responding yet" on timeout — no silent hang), (d) `gpsProvider`/`gpsDeviceId` written to the Unit record only on confirmed contact (or an explicit "save anyway" override). No schema migration — Sheets is schema-less.
3. **Live status → real `gpsStatus`.** On app load, `gpsFleetStatus()` populates an in-memory `gpsLive` map keyed by `deviceId`. Compute the displayed `gpsStatus` for a mapped unit from it (`online` → `Reporting`, stale/`Verify` heuristic → `Verify`, offline → `Not Reporting`), falling back to the stored `u.gpsStatus` field when the backend is unreachable. The existing pills (config `gpsStatus`: Reporting/Verify/Not Reporting) and the R3b "No GPS" data-chip stay as-is — only their *source* changes. Add the "as of `<ts>`, live link unavailable" notice on fallback (spec §7).
4. **Enriched GPS section (on open).** In the unit-detail GPS section, for a mapped unit: render a "last seen" line (timestamp + Google-maps link using the existing `GOOGLE_MAPS_KEY`) and an ignition-state chip from `gpsUnitStatus()`. Add a lightweight 30s refresh scoped to the open popup only (interval cleared on close), mirroring `useFleet.js` but single-unit. Distinguish "Not wired for ignition" from "Not Reporting" (spec §7).
5. **GPS status & alert history feed (spec §6a) — NEW step.** A read-only, chat-dock-styled (`.chat-feed` visual language, no compose box) scrollable timeline inside the GPS section, fetched via `gpsDeviceEvents()` on open. Renders `status_change`/`alert`/`shutdown_command` rows with icon + timestamp + severity color (existing status colors, no new palette). Empty state: "No GPS events yet." This becomes the destination for shutdown-command entries in step 6 below — **the generic unit History section (`historyFor()`) is explicitly NOT touched**, avoiding two competing audit trails (spec §7, revised).
6. **Remote shutdown control (Hapn only).**
   - **Role gate (AUTHORITY — resolve before building):** renders only for Owner + Dispatcher + Mechanic/M.Tech. The built-in role ids are `owner`/`admin`, `mechanic`/`mtech`, plus whatever the Dispatcher role id actually is in this deployment (there is no literal `dispatcher` in `BUILTIN_ROLE_TIERS` — **confirm the real role id with Jac / `settings.roleMeta` before wiring**, do not guess). Gate by an explicit allow-list of role ids, absent-from-DOM for others (not disabled) — matches the D5 "omit, don't hide" pattern used for cost/margin fields.
   - Renders only when `gpsProvider === 'Hapn'` (the other three are read-only).
   - Styled as the ignition-critical hold/release-to-arm hazard control (red variant), reusing the cancel-arc pattern (`.cancel-arc` in style.css).
   - On confirm → `gpsShutdown(deviceId)`; explicit success/failure toast, **no auto-retry**. The backend (A3c) writes the `device_events` row server-side on this call — no separate Rental Wrangler-side persistence step needed; step 5's feed picks it up on next open/refresh.
7. **Driving Score KPI.** Wire the `null` placeholder (app.js:~7887, "Driving Score = GPS backend") to real data where the provider exposes it (Bouncie trips carry the signals; others may stay `null`). Keep `null` where unavailable rather than faking a number.
8. **R-rulebook + gates.** `data-r` stamps on every new element (wizard steps, last-seen line, ignition chip, history feed rows, shutdown control). Map to existing rules where they fit, else add `RULE_META` rows in the same edit. Regenerate `node ci/gen-rule-usage.mjs`. **Wizard is a popup — check whether it needs a `WINDOW_CATALOG` entry** (the enriched GPS section itself stays inside the existing unit-detail window and needs none, but a standalone wizard popup likely does; confirm against `ci/check-window-catalog.mjs`'s actual trigger condition when building step 2). Update the code-map if a chapter banner was added (step 1).

## Test checkpoints

- **After step 2:** the wizard's live poll loop correctly progresses waiting→seen→confirmed against a mocked provider response, and shows the explicit timeout state (not a silent hang) when polling never resolves; `gpsProvider`/`gpsDeviceId` are only written on confirmed contact or explicit override, never on a bare form submit.
- **After step 3:** with a mocked fleet-status response, the Units list pills reflect live status; with the mock forced to fail, pills fall back to stored values + the "as of" notice — no silent stale-as-live.
- **After step 4:** opening a mapped unit renders last-seen + ignition; the 30s refresh updates and is cleared on close (no leaked interval); "Not wired for ignition" renders distinctly.
- **After step 5:** the history feed renders all three entry types from mocked `device_events` data, in order, with correct severity coloring; the empty state renders when the mock returns no rows.
- **After step 6:** the shutdown control is present only for the allow-listed roles AND only for Hapn units; hold-to-arm fires exactly one command; a mocked non-ack shows failure with no retry; a mocked ack causes a `shutdown_command` row to appear in the feed from step 5 on next refresh.
- **Before push:** all gates green — `node ci/smoke.mjs`, `node ci/logic-test.mjs`, `node ci/gen-rule-usage.mjs --check`, `node ci/check-window-catalog.mjs`, `node tools/gen-code-map.mjs --check`; zero R0 lint; screenshot the wizard, the enriched GPS section, and the history feed (seed data + mock backend) for a jactec-ui self-critique. Port 8000→9147 for browser gates, then `git checkout -- ci/`.
- **Staging E2E (promotion-time only):** against the real backend — live location renders for a mapped unit; the wizard successfully onboards a real test device end-to-end; the history feed shows a real status transition; remote shutdown tested against a **non-critical test tracker only**, never fleet equipment in business hours.

## Risks

- **Backend not ours yet at build time** — mitigated by building Track B against a mock; real wiring proven only at Staging E2E.
- **Safety (remote shutdown)** — role-gated (absent-from-DOM), Hapn-only, hold-to-arm, no auto-retry, audited via the history feed. The single highest-risk surface; its role gate is an explicit Jac decision, not inferred.
- **Dispatcher role id ambiguity** — the role framework's "Dispatcher" lens has no matching built-in role id; resolve the real id before gating (flagged in step 6) so the control isn't accidentally shown to the wrong role or hidden from the right one.
- **Direct-to-backend auth exposure** — the GPS backend is reached from the browser with the team token; confirm its own auth (README: `x-auth-token` on all `/api/*`) and CORS lock to our origins hold before go-live.
- **`app.js` size** — keep the new GPS client + render in one new APP chapter; update the code-map if a banner moves.
- **`device_events` is now the single audit trail** (spec §7, revised) — no longer split across the GPS backend's Postgres and a Sheets History log; simpler than the original plan, but means the history feed (step 5) is a hard dependency for anyone auditing a shutdown, not a nice-to-have. If A3c (the fork-side table/routes) slips, shutdown ships with no audit UI — don't decouple step 6 from step 5's dependency.
- **Wizard scope creep per provider** — Hapn's IMEI-entry step and the Deere/Yanmar/Bouncie picker step are genuinely different UIs; resist unifying them into one generic form just for tidiness, since the underlying capability differs (spec §4's provisioning-reality research).
- **Alert-history honesty** — provider alert endpoints return current state, not history; don't let the feed's UI imply a longer backfill exists than it does (spec §9).
